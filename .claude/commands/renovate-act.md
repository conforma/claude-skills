# Renovate Act — Execution Phase

Read a triage report generated by `/renovate-triage`, walk through categories interactively, re-validate each PR's current state, and execute approved actions (approve + auto-merge, or close with comment).

## Instructions

Parse `$ARGUMENTS` for optional flags:
- `--report <path>` — Path to a specific report JSON file. Defaults to the latest `renovate-triage-*.json` in `.claude/reports/`.
- `--category <name>` — Only process one category (abandoned_branch, superseded, security, go_version, major, routine, stale, needs_review).
- `--repo <owner/repo>` — Only process PRs in a specific repo.
- `--dry-run` — Show what actions would be taken without executing them.

## Step 1: Load the Triage Report

Find and load the report JSON:

```bash
# Find the latest report, or use --report path
ls -t .claude/reports/renovate-triage-*.json 2>/dev/null | head -1
```

If no report is found, tell the user:
> "No triage report found in .claude/reports/. Run /renovate-triage first to generate one."

Read the JSON file into memory.

### Staleness Check

Calculate the age of the report from `generated_at`. If older than 24 hours:
> "⚠️ This report is {n} hours old. PR states may have changed. Consider re-running /renovate-triage first. Continue anyway? (y/n)"

If the user declines, stop.

### Resumability

Count PRs where `outcome` is non-null (already processed in a previous run). If any:
> "Skipping {n} PRs already processed in a previous run. {m} PRs remaining."

Filter those PRs out of the working set.

### Apply Filters

If `--category` was specified, keep only that category.
If `--repo` was specified, keep only PRs matching that repo.

Report the filtered scope:
> "Processing {n} PRs across {categories} in {repos}."

## Step 2: Walk Categories in Priority Order

Process categories in this order: abandoned_branch, superseded, security, go_version, major, routine, stale, needs_review.

Skip empty categories silently.

### For Batch-Safe Categories (abandoned_branch, superseded, routine)

Present the category:
```
## Category: {name} ({n} PRs) — Recommend: {action}

{reasoning from the report}

| # | PR | Repo | Branch | Title | Age |
|---|-----|------|--------|-------|-----|
| 1 | #{number} | {repo} | {branch} | {title} | {age}d |
| 2 | ... |
...

Approve this batch? (y / skip / review individually)
```

- **If "y"**: Process all PRs in this batch (proceed to Step 3 for each).
- **If "skip"**: Skip this entire category.
- **If "review individually"**: Present each PR one at a time with "Action? (approve / close / skip)".

### For Review Categories (security, go_version, major, stale, needs_review)

Present each PR individually:

```
### PR #{number} — {repo}
- **Title:** {title}
- **Branch:** {base_branch}
- **Age:** {age} days
- **Author:** {author}
- **CI Status:** {ci_status}
- **Recommended action:** {recommended_action}
- **Reason:** {reason}
{If go_version: "- **Go image verification:** {go_verification}"}
{If in cascade group: "- **Cascade group:** Part of {n}-PR cascade on {repo}/{branch}"}

Action? (approve / close / skip)
```

- **If "approve"**: Proceed to Step 3 with action = approve.
- **If "close"**: Proceed to Step 3 with action = close.
- **If "skip"**: Mark as skipped, continue.

**Note for security category:** Although the default action is approve + auto-merge, present each security PR individually since the user expressed that they don't want to miss important security issues.

## Step 3: Re-Validate Before Action (MANDATORY)

**This step is a hard architectural constraint. There is no flag to bypass it.**

Before executing ANY action on a PR, fetch its current state:

```bash
gh pr view <NUMBER> --repo <REPO> --json state,mergeable,statusCheckRollup,commits,updatedAt
```

Validate:
1. **PR is still open**: `state` must be `"OPEN"`. If merged or closed, skip:
   > "PR #{number} is already {state}. Skipping."

2. **No new commits**: Compare the latest commit SHA or `updatedAt` against what's in the report. If changed:
   > "PR #{number} has new activity since the triage report. Skipping — re-run /renovate-triage for updated data."

3. **CI status** (for approve/merge actions only): Check `statusCheckRollup`. If CI is failing:
   > "PR #{number} has failing CI checks. Skipping merge — review CI failures manually."
   If CI is pending, note it:
   > "PR #{number} has pending CI checks. Auto-merge will wait for CI to complete."

If validation fails, record `outcome: "skipped"` with the reason. Continue to next PR.

## Step 4: Execute Action

### If --dry-run

Do NOT execute any action. Instead, print:
```
[DRY RUN] Would {action}: {repo}#{number} ({title})
  Reason: {reason}
  {If close: "Comment: {close comment template}"}
```

Record `outcome: "dry_run"` in the report.

### If approve + auto-merge

```bash
gh pr review <NUMBER> --repo <REPO> --approve --body "Approved by automated triage: {reason}"
gh pr merge <NUMBER> --repo <REPO> --auto
```

If either command fails, log the error and record `outcome: "failed"`.
If both succeed, record `outcome: "merged"` (meaning auto-merge was enabled, not necessarily merged yet).

### If close

Use the close comment template from SKILL.md based on the PR's category. Fill in the template variables ({branch}, {newer_pr}, {age}).

```bash
gh pr close <NUMBER> --repo <REPO> --comment "<filled close comment>"
```

If the command fails, log the error and record `outcome: "failed"`.
If it succeeds, record `outcome: "closed"`.

## Step 5: Update Both Report Files

After processing each PR (or batch), update both report files:

### Update JSON

Read the JSON file, find the PR entry by `(repo, number)`, set:
- `outcome`: "merged", "closed", "skipped", "failed", or "dry_run"
- `outcome_at`: current ISO 8601 timestamp

Write the updated JSON back to the same file.

### Update Markdown

Read the markdown file. Find the table row for this PR (match by PR number and repo). Append the outcome to the Action column:
- `✅ Merged` (auto-merge enabled)
- `❌ Closed`
- `⏭️ Skipped ({reason})`
- `💥 Failed ({error})`
- `👀 Dry run`

Write the updated markdown back to the same file.

## Step 6: Final Summary

After processing all categories, present a summary:

```
## Triage Act Complete

| Action | Count |
|--------|-------|
| Approved + auto-merge | {n} |
| Closed | {n} |
| Skipped | {n} |
| Failed | {n} |
| Dry run | {n} |
| Remaining (not processed) | {n} |

{If any failures:}
### Failures
- {repo}#{number}: {error message}

{If any remaining:}
Run /renovate-act again to process remaining PRs.

Reports updated:
- .claude/reports/renovate-triage-{date}.json
- .claude/reports/renovate-triage-{date}.md
```
